# 日志和错误规范

## 日志级别定义

**日志级别使用原则**：

### Debug - 调试信息
- 用于开发阶段排查问题
- 包含详细的方法调用、参数值、中间状态
- 生产环境通常关闭

### Info - 常规信息
- 记录系统正常运行状态
- 重要的业务操作、状态变更
- 用户登录、关键数据更新等

### Warn - 警告信息
- 表示潜在问题但系统仍可运行
- 非关键错误、降级处理、配置问题
- 资源不足、性能警告等

### Error - 错误信息
- 表示功能失败但系统可继续运行
- 业务逻辑错误、外部服务调用失败
- 数据验证失败、权限不足等

### Fatal - 致命错误
- 系统无法继续运行
- 关键组件初始化失败、内存耗尽
- 数据库连接失败等

## 日志消息格式规范

**基本格式**：`message` + `context object`

**组件说明**：
- **message**：简洁描述发生了什么（使用现在时态）
- **context**：匿名对象包含相关参数和状态信息
- **exception**（可选）：异常对象作为第二个参数传递

**原则**：
- 消息文本保持简洁，不包含变量插值
- 所有上下文信息通过匿名对象传递
- 异常始终作为单独参数传递

**示例**：

```csharp
// Debug 级别
Log.Debug("[Player] Player logged in", new { PlayerId = playerId, PlayerName = playerName });

// Info 级别
Log.Info("[Player] Player joined the game", new { PlayerId = playerId, Position = position });

// Warning 级别
Log.Warn("[Resource] Failed to load asset", new { AssetPath = assetPath, Reason = error.Message });

// Error 级别
Log.Error("[Network] Lost connection to server", exception, new { ServerAddress = serverAddress });
```

## 错误信息语法规范

**错误消息结构**：`What happened? Why it happened? How to fix?`

**组件说明**：
- **What**：发生了什么错误
- **Why**：错误的原因（如果知道）
- **How**：如何解决或下一步操作

**示例**：

```csharp
// 验证错误
throw new ArgumentException("Player name cannot be empty. Name validation failed. Please provide a valid player name.");

// 业务逻辑错误
throw new InvalidOperationException("Cannot attack while dead. Player health is 0. Revive player before attacking.");

// 资源错误
throw new FileNotFoundException("Asset file not found. File path is invalid or file does not exist. Check the asset path and ensure the file exists.");
```

## 异常处理规范

### 异常捕获原则

1. **只捕获你能处理的异常**
2. **使用具体的异常类型**，避免捕获通用 Exception
3. **记录异常上下文信息**
4. **重新抛出时保留原始异常**

### 异常处理示例

```csharp
public void LoadPlayerData(int playerId)
{
    try
    {
        var data = _dataService.GetPlayerData(playerId);
        // 处理数据
    }
    catch (DataServiceException ex)
    {
        // 记录详细的错误信息
        Log.Error("[PlayerService] Failed to load player data", ex, new { PlayerId = playerId });

        // 抛出业务异常，保留原始异常
        throw new PlayerDataLoadException($"Failed to load player data for {playerId}", ex);
    }
    catch (TimeoutException ex)
    {
        // 处理超时异常
        Log.Warn("[PlayerService] Timeout loading player data", new { PlayerId = playerId });
        throw;
    }
}
```

## 自定义异常规范

### 命名规则
`XxxException`

### 构造函数要求
- 提供有意义的错误消息
- 支持传递内部异常
- 可选的错误代码或上下文数据

### 自定义异常示例

```csharp
public class PlayerNotFoundException : Exception
{
    public int PlayerId { get; }

    public PlayerNotFoundException(int playerId)
        : base($"Player with ID {playerId} was not found. The player may not exist or has been deleted.")
    {
        PlayerId = playerId;
    }

    public PlayerNotFoundException(int playerId, Exception innerException)
        : base($"Player with ID {playerId} was not found. The player may not exist or has been deleted.", innerException)
    {
        PlayerId = playerId;
    }
}

public class InsufficientResourceException : Exception
{
    public string ResourceType { get; }
    public int Required { get; }
    public int Available { get; }

    public InsufficientResourceException(string resourceType, int required, int available)
        : base($"Insufficient {resourceType}. Required: {required}, Available: {available}. " +
               $"Please acquire more {resourceType} or reduce the requirement.")
    {
        ResourceType = resourceType;
        Required = required;
        Available = available;
    }
}
```

## 异常处理最佳实践

### 应该做的

1. **使用具体的异常类型**
   ```csharp
   // Good: 捕获具体异常
   catch (FileNotFoundException ex)
   {
       Log.Error("[FileLoader] File not found", ex, new { FilePath = filePath });
   }
   ```

2. **记录异常上下文信息**
   ```csharp
   catch (Exception ex)
   {
       Log.Error("[Service] Operation failed", ex, new { UserId = userId, ActionName = actionName });
   }
   ```

3. **重新抛出时保留原始异常**
   ```csharp
   catch (IOException ex)
   {
       throw new ServiceException("Failed to load data", ex);
   }
   ```

### 不应该做的

1. **捕获所有异常**
   ```csharp
   // Bad: 捕获所有异常会掩盖问题
   catch (Exception ex)
   {
       // 吞噬所有异常
   }
   ```

2. **不记录就重新抛出**
   ```csharp
   // Bad: 重新抛出但未记录任何信息
   catch
   {
       throw;
   }
   ```

3. **空的 catch 块**
   ```csharp
   // Bad: 空的 catch 块
   catch (Exception)
   {
       // 什么都不做
   }
   ```

## 使用规范

### 何时使用日志

日志的核心价值在于**复盘**。它记录了程序在运行时的“心路历程”，主要用于那些不需要程序停止运行，但开发者需要知道发生了什么的情况。

- **关键状态迁移：** 记录重要的逻辑节点。
  - *例子：* 玩家进入副本、开始匹配、存档成功、NPC 行为树切换状态。
- **非致命性错误：** 某些资源加载失败或配置缺失，但有默认回退方案。
  - *例子：* “找不到音效文件 `Explosion_01`，改用默认音效。”（使用 `LogWarning`）
  ```csharp
  Log.Warn("[Audio] Sound effect not found, using default", new { SoundName = soundName });
  ```
- **性能监控与埋点：** 记录执行时长或玩家行为。
  - *例子：* 统计加载场景耗时、记录玩家点击商城按钮的频率。
  ```csharp
  Log.Info("[Scene] Scene loaded successfully", new { SceneName = sceneName, LoadTime = elapsedMs });
  Log.Debug("[UI] Button clicked", new { Button = "Shop", Count = clickCount });
  ```
- **异步流程追踪：** 在复杂的网络通信或异步加载中，记录请求与回调的匹配情况。

**建议习惯：**  **区分等级：** 严格使用 `Info` (日常)、`Warning` (潜在风险)、`Error` (逻辑错误)。

- **结构化：** 包含模块前缀（如 `[Network]`, `[Battle]`），方便在 Console 中过滤。

### 何时使用异常 

异常的核心价值在于**保护**。它用于处理那些“不该发生且无法继续”的状况。一旦抛出异常，当前的执行流会被打断，强制开发者或系统介入。

- **违反前置条件（契约）：** 当函数的参数完全非法，继续执行会导致逻辑崩溃时。
  - *例子：* `Damage(int amount)` 接收到了负数。
- **硬件或环境失效：** 无法绕过的外部错误。
  - *例子：* 核心配置文件损坏（Luban 表解析失败）、内存耗尽、服务器连接被物理切断。
- **逻辑上的“不可能”路径：** 在 `switch-case` 的 `default` 分支中，如果理论上所有情况都应被覆盖，则抛出异常。
  - *例子：* 一个枚举只有三种状态，但在逻辑判断中出现了第四种。
- **框架/工具链的硬限制：** 确保其他协作者遵守开发规范。
  - *例子：* 某个脚本必须挂载在具有 `Rigidbody` 的物体上，否则在 `Awake` 中抛出异常。

### 对比总结：日志 vs 异常

| **维度**         | **日志 (Logging)**              | **异常 (Exception)**              |
| ---------------- | ------------------------------- | --------------------------------- |
| **程序影响**     | 不影响流程，代码继续执行        | 打断当前流程，跳转至 Catch 或崩溃 |
| **处理目标**     | 开发者（事后分析）              | 程序逻辑（运行时防御）            |
| **典型性能开销** | 较低（尤其在 Release 版关闭后） | 较高（涉及堆栈追踪的捕获）        |
| **关注点**       | “刚才发生了什么？”              | “这里出大事了，不能再跑了！”      |
