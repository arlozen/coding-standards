# 类成员排列规范

## 基本原则

类成员排列的目标不是“机械排序”，而是让阅读者按最短路径理解类型职责、公共入口和实现细节。

统一遵循以下原则：

1. 同一类中优先按**职责分组**，再按**可见性**排序。
2. 对外契约优先于内部实现细节。
3. Unity 类要优先体现组件的主要使用方式，而不是套用单一模板。
4. 同一类型内保持稳定顺序，不因单个成员的访问级别频繁穿插。

**默认访问修饰符顺序**：`public` → `protected` → `internal` → `private`

## 普通类成员排列顺序

### 推荐顺序

普通类从上到下推荐按以下顺序排列：

1. 常量（`const`）
2. 静态只读字段（`static readonly`）
3. 静态字段（`static`）
4. 实例字段
5. 构造函数
6. 属性与事件
7. 公共方法
8. 受保护方法
9. 显式接口实现
10. 私有方法
11. 局部辅助类型

### 说明

- 字段按“静态在前，实例在后”排列。
- 属性与事件都属于类型的对外状态/通知接口，应放在公共行为之前。
- 显式接口实现应视为**单独分组**，不要与普通 `public` 方法交错。
- 私有方法放在实现细节区域，避免打断对外 API 的阅读流。

### 显式实现接口规范

显式实现接口成员时，统一遵循以下规则：

1. 显式接口实现单独成组，放在公共/受保护成员之后、私有实现之前。
2. 不要把显式接口实现穿插到同名公共方法之间。
3. 同一个接口的显式实现连续摆放；多个接口按接口名分组。
4. 如果同时提供公共方法和显式接口实现，先展示公共 API，再展示显式实现。
5. 显式接口实现的主要目的是表达“这是契约适配层”，因此它应靠近对外接口区域，但不应混入主要公共 API。

### 普通类示例

```csharp
public sealed class Example : IDisposable, IResettable
{
    public const int MaxCount = 100;
    private const string DefaultName = "Unknown";

    public static readonly Example Empty = new(0, DefaultName);
    private static readonly Random _rng = new();

    private static int s_globalCounter;

    private int _id;
    private string _name;

    public Example(int id, string name)
    {
        _id = id;
        _name = name;
    }

    public int Id => _id;

    public string Name
    {
        get => _name;
        set => _name = value ?? DefaultName;
    }

    public static Example CreateRandom()
    {
        return new Example(_rng.Next(MaxCount), "Random");
    }

    public void Dispose()
    {
        ResetCore();
    }

    void IResettable.Reset()
    {
        ResetCore();
    }

    private void ResetCore()
    {
        _id = 0;
        _name = DefaultName;
    }
}
```

## Unity 类成员排列顺序

Unity 组件不是单一形态。成员顺序必须反映组件的主导使用方式。

### 通用固定顺序

无论是哪种 Unity 组件，以下分组保持一致：

1. 常量（`const`）
2. 静态只读字段（`static readonly`）
3. 静态字段（`static`）
4. 序列化字段（`[SerializeField] private`）
5. 非序列化实例字段
6. 属性与事件
7. 其余成员按组件类型选择排序模板

**序列化字段规范**：

- 序列化字段优先使用 `[SerializeField] private`。
- 如果需要外部访问，通过属性或方法暴露。
- 不要为了 Inspector 可见性直接放宽字段访问级别。

### 模板 A：自驱动组件

适用于以下场景：

- 组件主要由 Unity 生命周期驱动。
- `Update`、`FixedUpdate`、`OnEnable` 等是阅读主入口。
- 外部调用只是补充能力，而不是主要用法。

**推荐顺序**：

1. 常量 / 静态字段
2. 序列化字段
3. 非序列化实例字段
4. 属性与事件
5. Unity 生命周期方法
6. 公共方法
7. 受保护方法
8. 显式接口实现
9. 私有方法
10. 协程

### 模板 B：外部 API 组件

适用于以下场景：

- 组件主要作为服务、门面、控制器或可复用能力提供者。
- 外部代码通过公共方法驱动它。
- 生命周期更多承担初始化、注册、清理等内部职责。

**推荐顺序**：

1. 常量 / 静态字段
2. 序列化字段
3. 非序列化实例字段
4. 属性与事件
5. 公共方法
6. 受保护方法
7. Unity 生命周期方法
8. 显式接口实现
9. 私有方法
10. 协程

### 选择规则

- 如果阅读该组件时最先需要理解的是“它在 Unity 帧循环里怎么工作”，使用**模板 A**。
- 如果阅读该组件时最先需要理解的是“外部如何调用它”，使用**模板 B**。
- 同一目录或同一类族已有稳定模式时，优先保持局部一致性，不要孤立地切换模板。

### Unity 生命周期方法内部顺序

生命周期方法内部仍然按 Unity 实际调用阶段排序，推荐如下：

1. 编辑器与校验阶段：`Reset`、`OnValidate`
2. 初始化阶段：`Awake`、`OnEnable`、`Start`
3. 帧循环阶段：`FixedUpdate`、`Update`、`LateUpdate`
4. 应用状态阶段：`OnApplicationFocus`、`OnApplicationPause`、`OnApplicationQuit`
5. 结束阶段：`OnDisable`、`OnDestroy`

如果某些生命周期不存在，直接跳过，不要为了“占位”制造空方法。

### Unity 示例 A：自驱动组件

```csharp
using System.Collections;
using UnityEngine;

public class PlayerController : MonoBehaviour
{
    private const float MaxHealth = 100f;
    private static readonly Vector3 DefaultSpawn = Vector3.zero;

    [SerializeField] private float _moveSpeed = 5f;
    [SerializeField] private Rigidbody2D _rigidbody;
    [SerializeField] private float _maxHealth = 100f;

    private float _health;
    private Vector2 _input;

    public bool IsAlive => _health > 0f;

    public float MoveSpeed
    {
        get => _moveSpeed;
        set => _moveSpeed = Mathf.Max(0f, value);
    }

    public float MaxHealthValue => _maxHealth;

    private void Awake()
    {
        _health = MaxHealth;
    }

    private void OnEnable()
    {
        // Register events.
    }

    private void Start()
    {
        Spawn(DefaultSpawn);
    }

    private void FixedUpdate()
    {
        Move();
    }

    private void Update()
    {
        HandleInput();
    }

    private void OnDisable()
    {
        // Unregister events.
    }

    private void OnDestroy()
    {
        // Release resources.
    }

    public void TakeDamage(float amount)
    {
        _health = Mathf.Max(0f, _health - amount);
    }

    private void HandleInput()
    {
        _input = new Vector2(Input.GetAxis("Horizontal"), Input.GetAxis("Vertical"));
    }

    private void Move()
    {
        _rigidbody.velocity = _input * _moveSpeed;
    }

    private void Spawn(Vector3 position)
    {
        transform.position = position;
    }

    private IEnumerator RespawnCoroutine()
    {
        yield return new WaitForSeconds(3f);
        Spawn(DefaultSpawn);
        _health = MaxHealth;
    }
}
```

### Unity 示例 B：外部 API 组件

```csharp
using UnityEngine;

public class CameraShakeService : MonoBehaviour, ICameraShakeService
{
    [SerializeField] private Camera _targetCamera;
    [SerializeField] private AnimationCurve _intensityCurve;

    private float _duration;
    private float _elapsedTime;
    private bool _isPlaying;

    public bool IsPlaying => _isPlaying;

    public void Play(float duration)
    {
        _duration = duration;
        _elapsedTime = 0f;
        _isPlaying = true;
    }

    public void Stop()
    {
        _duration = 0f;
        _elapsedTime = 0f;
        _isPlaying = false;
        _targetCamera.transform.localPosition = Vector3.zero;
    }

    private void Awake()
    {
        _targetCamera.transform.localPosition = Vector3.zero;
    }

    private void Update()
    {
        if (!_isPlaying)
            return;

        _elapsedTime += Time.deltaTime;
        UpdateShake();
    }

    private void OnDisable()
    {
        Stop();
    }

    void ICameraShakeService.Play(float duration)
    {
        Play(duration);
    }

    private void UpdateShake()
    {
        if (_elapsedTime >= _duration)
        {
            Stop();
            return;
        }

        var progress = _elapsedTime / _duration;
        var intensity = _intensityCurve.Evaluate(progress);
        _targetCamera.transform.localPosition = Random.insideUnitSphere * intensity;
    }
}
```

## 补充要求

1. 不要因为成员很少就打乱顺序；小类同样保持规范。
2. 当部分类别不存在时，直接省略该分组，不保留空白占位。
3. 同组内优先保持语义聚合，例如同一接口实现、同一生命周期阶段、同一类辅助方法尽量放在一起。
4. 如果现有文件已有稳定且清晰的排列方式，修改时优先做增量对齐，不要为“绝对标准化”制造大面积无收益重排。
